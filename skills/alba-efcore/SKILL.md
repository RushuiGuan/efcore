# Albatross EF Core Patterns

This skill documents how to correctly use the `Albatross.EFCore` library. Follow these patterns precisely.

---

## Architecture Overview

The data access layer is organized in layers:

```
Controller / Command Handler
        ↓ (calls SaveChanges once)
    Service
        ↓ (depends on)
   Repository
        ↓ (depends on)
   IDbSession (DbContext wrapper)
```

**Key constraint:** Services depend on Repositories. Services never depend on `IDbSession` directly.

---

## 1. Entities

Entities are plain C# classes. Co-locate the entity class and its `EntityMap` in the same file.

```csharp
public class Company {
    public required Guid Id { get; init; }
    [MaxLength(256)] public required string Name { get; set; }
    public string? Description { get; set; }

    // Navigation properties for child aggregates
    public List<Contact> Contacts { get; init; } = [];
}

public class CompanyEntityMap : EntityMap<Company> {
    public override void Map(EntityTypeBuilder<Company> builder) {
        builder.HasKey(x => x.Id);
        builder.Property(x => x.Id).ValueGeneratedNever(); // always — Guid PKs are assigned by the app
        builder.HasIndex(x => x.Name).IsUnique();          // enforce uniqueness at DB level
        builder.HasMany(x => x.Contacts)
            .WithOne(x => x.Company)
            .HasForeignKey(x => x.CompanyId);
    }
}
```

**Rules:**
- `Guid` PKs always use `.ValueGeneratedNever()` — the application assigns the ID
- Define unique constraints in `EntityMap`, not with data annotations
- Never use `[ForeignKey]` or `[InverseProperty]` — configure relationships in `EntityMap`
- String columns default to `varchar` (non-unicode). Apply `[Unicode(true)]` only when truly needed

## 2. DbSession

Define a session interface and class per module/bounded context.

```csharp
public interface ICrmDbSession : IDbSession { }

public class CrmDbSession : DbSession, ICrmDbSession {
    public CrmDbSession(DbContextOptions option) : base(option) { }

    protected override void OnModelCreating(ModelBuilder modelBuilder) {
        modelBuilder.HasDefaultSchema(Constants.Schema);
        modelBuilder.BuildEntityModels(); // auto-registers all EntityMap classes in the assembly
    }
}
```

`modelBuilder.BuildEntityModels()` is generated by the `Albatross.EFCore.CodeGen` source generator. It must be explicitly referenced as a NuGet package in the project file. When referenced, it auto-discovers all `EntityMap<T>` subclasses in the assembly — no manual registration needed.

If `Albatross.EFCore.CodeGen` is not referenced, call each entity map manually instead:

```csharp
protected override void OnModelCreating(ModelBuilder modelBuilder) {
    modelBuilder.HasDefaultSchema(Constants.Schema);
    new CompanyEntityMap().Build(modelBuilder);
    new ContactEntityMap().Build(modelBuilder);
    // one line per entity map
}
```

Do not use `DbSessionWithEventHandlers` — it is being redesigned and should not be used in new code.

## 3. Repository

Create a repository for each **aggregate root** only — not for every entity.

```csharp
public interface ICompanyRepository : IRepository {
    Task<List<Company>> GetAll(CancellationToken cancellationToken);
    Task<Company> GetById(Guid id, CancellationToken cancellationToken);
    Task<Company?> GetByName(string name, CancellationToken cancellationToken);
}

public class CompanyRepository : Repository<ICrmDbSession>, ICompanyRepository {
    public CompanyRepository(ICrmDbSession session) : base(session) { }

    public override bool IsUniqueConstraintViolation(Exception err) =>
        SqlServerExt.IsUniqueConstraintViolation(err) || PostgresExt.IsUniqueConstraintViolation(err);

    public override bool IsForeignKeyConstraintViolation(Exception err) =>
        SqlServerExt.IsForeignKeyConstraintViolation(err) || PostgresExt.IsForeignKeyConstraintViolation(err);

    public Task<List<Company>> GetAll(CancellationToken cancellationToken) =>
        session.DbContext.Set<Company>().ToListAsync(cancellationToken);

    public async Task<Company> GetById(Guid id, CancellationToken cancellationToken) {
        var entity = await session.DbContext.Set<Company>()
            .Include(x => x.Contacts)  // always eager load — never rely on lazy loading
            .FirstOrDefaultAsync(x => x.Id == id, cancellationToken);
        if (entity == null) throw new NotFoundException<Company>(id);
        return entity;
    }

    public Task<Company?> GetByName(string name, CancellationToken cancellationToken) =>
        session.DbContext.Set<Company>()
            .FirstOrDefaultAsync(x => x.Name == name, cancellationToken);
}
```

**Rules:**
- **Always eager load** — use `.Include()` / `.ThenInclude()` explicitly. Never enable lazy loading.
- **Throw `NotFoundException<T>` when querying by key** and the entity is not found. Never return `null` from a by-key lookup.
- Nullable return (`Company?`) is acceptable for lookups that are not by key (e.g. by name), where "not found" is a valid outcome the caller handles.
- The repository holds the `IDbSession` but **does not call `SaveChanges`** — that is `IRepository.SaveChangesAsync`, called by the caller (controller/command handler).

**Constraint violation detection** — SQL Server-specific detection lives in `Albatross.EFCore.SqlServer` and PostgreSQL-specific detection lives in `Albatross.EFCore.PostgreSQL`. Implement `IsUniqueConstraintViolation` and `IsForeignKeyConstraintViolation` using the extension methods from whichever of those packages the project references:

```csharp
// Project references both Albatross.EFCore.SqlServer and Albatross.EFCore.PostgreSQL:
using SqlServerExt = Albatross.EFCore.SqlServer.Extensions;
using PostgresExt = Albatross.EFCore.PostgreSQL.Extensions;

public override bool IsUniqueConstraintViolation(Exception err) =>
    SqlServerExt.IsUniqueConstraintViolation(err) || PostgresExt.IsUniqueConstraintViolation(err);

// Project references only Albatross.EFCore.PostgreSQL:
public override bool IsUniqueConstraintViolation(Exception err) =>
    PostgresExt.IsUniqueConstraintViolation(err);
```

### NotFoundException

```csharp
// Use the generic form — it uses typeof(T).Name as the entity name automatically
throw new NotFoundException<Company>(id);           // "Company (guid) is not found"
throw new NotFoundException<Company>(name);         // "Company (name) is not found"
```

## 4. Service

Services contain business logic and depend on repositories.

```csharp
public interface ICompanyService {
    Task<Company> Create(Guid id, string name, CancellationToken cancellationToken);
    Task<Company> Update(Guid id, string name, CancellationToken cancellationToken);
    Task Delete(Guid id, CancellationToken cancellationToken);
}

public class CompanyService : ICompanyService {
    readonly ICompanyRepository companyRepository;

    public CompanyService(ICompanyRepository companyRepository) {
        this.companyRepository = companyRepository;
    }

    public async Task<Company> Create(Guid id, string name, CancellationToken cancellationToken) {
        var company = new Company { Id = id, Name = name };
        companyRepository.Add(company);
        return company;
        // Note: no SaveChangesAsync here — caller is responsible
    }

    public async Task<Company> Update(Guid id, string name, CancellationToken cancellationToken) {
        var company = await companyRepository.GetById(id, cancellationToken); // throws if not found
        company.Name = name;
        return company;
        // Note: no SaveChangesAsync here — EF change tracking handles the update
    }

    public async Task Delete(Guid id, CancellationToken cancellationToken) {
        var company = await companyRepository.GetById(id, cancellationToken);
        companyRepository.Delete(company);
        // Note: no SaveChangesAsync here
    }
}
```

**Rules:**
- Services **never depend on `IDbSession`** — only on repository interfaces
- Services **never call `SaveChangesAsync`** — that is the caller's responsibility
- Services **return entities**, not DTOs — transformation to view models happens at a higher layer
- Services **never pre-check for uniqueness** (e.g., "does a company with this name already exist?") — let the DB enforce it via unique constraint. The `SaveResults` from `SaveChangesAsync` tells the caller whether there was a `NameConflict` or `ForeignKeyConflict`.

## 5. Application Boundary: Saving

`SaveChangesAsync` is called **once**, at the controller or command handler level, after all service calls in a unit of work are complete.

```csharp
// In a controller or command handler:
await service.Create(id, name, cancellationToken);
var result = await repository.SaveChangesAsync(throwException: false, cancellationToken);

if (!result.Success) {
    if (result.NameConflict) return Conflict("Name already exists");
    if (result.ForeignKeyConflict) return UnprocessableEntity("Referenced entity not found");
    throw result.Error!;
}
```

`SaveResults` properties:
- `Success` — true if no exception
- `NameConflict` — unique constraint violation
- `ForeignKeyConflict` — foreign key constraint violation
- `Error` — the underlying exception (non-null when `!Success`)

Pass `throwException: true` when you want exceptions to propagate directly (e.g., in background jobs).

## 6. DI Registration

Register the session and repositories in a module-level extension method:

```csharp
public static class Extensions {
    public static IServiceCollection AddCrmDbSession(this IServiceCollection services) {
        services
            .AddScoped<ICrmDbSession>(provider => provider.GetRequiredService<CrmDbSession>())
            .AddScoped<ICompanyRepository, CompanyRepository>()
            .AddScoped<ICompanyService, CompanyService>();
        return services;
    }
}
```

Then register the database provider separately (typically based on config):

```csharp
// SQL Server
services.AddSqlServerWithContextPool<CrmDbSession>(
    provider => provider.GetRequiredService<ICrmConfig>().ConnectionString);

// PostgreSQL
services.AddPostgresWithContextPool<CrmDbSession>(
    provider => provider.GetRequiredService<ICrmConfig>().ConnectionString);
```

Use `WithContextPool` variants for production — they're more efficient. Use the non-pool variants (`AddSqlServer<T>`, `AddPostgres<T>`) only when the DbContext constructor has parameters beyond `DbContextOptions`.

## 7. Eager Loading

**Never** enable lazy loading. Always use `.Include()` and `.ThenInclude()` explicitly when related data is needed.

```csharp
// Load company with all contacts and their addresses
session.DbContext.Set<Company>()
    .Include(x => x.Contacts)
        .ThenInclude(c => c.Addresses)
    .FirstOrDefaultAsync(x => x.Id == id, cancellationToken);
```

If a repository method should return related data, include it in the query. If it shouldn't, don't include it — don't rely on the navigation property being populated.

## 8. Admin CLI (Database Migration)

Use `Albatross.EFCore.Admin` to create a standalone CLI tool for managing database migrations. This library provides three generic command handlers that plug directly into the `Albatross.CommandLine` framework. For the CommandLine patterns (`CommandHost`, `[Verb<>]`, `BaseHandler<T>`, etc.), consult the **alba-cli** skill.

### Migration DbContext classes

Create a migration-specific subclass per DB provider. These are used only by the EF tooling and the admin CLI — not by the application at runtime.

```csharp
// SQL Server migration context
public class CrmSqlServerMigration : CrmDbSession {
    public CrmSqlServerMigration() : this("any") { }
    public CrmSqlServerMigration(string connectionString)
        : base(Albatross.EFCore.SqlServer.Extensions.BuildMigrationOption<CrmDbSession>(
            Constants.Schema, connectionString)) { }
}

// PostgreSQL migration context
public class CrmPostgresMigration : CrmDbSession {
    public CrmPostgresMigration() : this("any") { }
    public CrmPostgresMigration(string connectionString)
        : base(Albatross.EFCore.PostgreSQL.Extensions.BuildMigrationOption<CrmDbSession>(
            Constants.Schema, connectionString)) { }
}
```

The parameterless constructor (`this("any")`) is required by the `dotnet ef` tooling at design time.

Only create migration context classes for the providers the application supports. If the app is PostgreSQL-only, only create `CrmPostgresMigration`.

### Built-in command handlers

`Albatross.EFCore.Admin` provides three generic handlers. Bind each to a command verb via `[Verb<>]` attributes:

| Handler | Purpose |
|---|---|
| `EFMigrate<T>` | Apply pending EF migrations (`MigrateAsync`) |
| `GenerateSqlScript<T>` | Print the CREATE script for the schema; optionally write to file and generate a DROP script |
| `ExecuteDeploymentScripts<T>` | Run `.sql` files from a versioned subdirectory of a given folder |

### Verbs.cs — wire commands to providers

```csharp
// PostgreSQL commands
[assembly: Verb<EFMigrationParams, EFMigrate<CrmPostgresMigration>>("postgres ef-migrate", Description = "Apply pending EF migrations to PostgreSQL")]
[assembly: Verb<GenerateSqlScriptParams, GenerateSqlScript<CrmPostgresMigration>>("postgres create-script", Description = "Generate SQL script for PostgreSQL")]
[assembly: Verb<ExecuteDeploymentScriptsParams, ExecuteDeploymentScripts<CrmPostgresMigration>>("postgres exec-script", Description = "Execute PostgreSQL deployment scripts")]

// SQL Server commands (omit if app is PostgreSQL-only)
[assembly: Verb<EFMigrationParams, EFMigrate<CrmSqlServerMigration>>("sqlserver ef-migrate", Description = "Apply pending EF migrations to SQL Server")]
[assembly: Verb<GenerateSqlScriptParams, GenerateSqlScript<CrmSqlServerMigration>>("sqlserver create-script", Description = "Generate SQL script for SQL Server")]
[assembly: Verb<ExecuteDeploymentScriptsParams, ExecuteDeploymentScripts<CrmSqlServerMigration>>("sqlserver exec-script", Description = "Execute SQL Server deployment scripts")]
```

### ParentParams.cs — command group with aliases

```csharp
[Verb("sqlserver", Alias = ["sql"], Description = "SQL Server commands")]
[Verb("postgres",  Alias = ["pg"],  Description = "PostgreSQL commands")]
public class ParentParams { }
```

### Program.cs — conditional DI based on command prefix

Register the correct DB provider and `IExecuteScriptFile` implementation based on which command group is invoked:

```csharp
static async Task<int> Main(string[] args) {
    await using var host = new CommandHost("MyApp Admin")
        .RegisterServices(RegisterServices)
        .AddCommands()
        .Parse(args)
        .WithConfig()
        .ConfigureHost(builder => {
            builder.UseSerilog();
            builder.ConfigureLogging((context, logging) => {
                var setupSerilog = new SetupSerilog();
                setupSerilog.UseConfigFile(EnvironmentSetting.DOTNET_ENVIRONMENT.Value, null, null, true);
                setupSerilog.Create();
            });
        })
        .Build();
    return await host.InvokeAsync();
}

static void RegisterServices(ParseResult result, IServiceCollection services) {
    services.AddCrmDbSession();  // registers ICrmDbSession, repositories, services
    var key = result.CommandResult.Command.GetCommandKey();
    if (key.StartsWith("sqlserver")) {
        services.AddConfig<ICrmConfig, Crm.Models.SqlServer.CrmConfig>();
        services.AddSingleton<IExecuteScriptFile, ExecuteSqlServerScriptFile>(); // handles GO batches
        services.AddScoped(p => new CrmSqlServerMigration(p.GetRequiredService<ICrmConfig>().ConnectionString));
        services.AddSqlServerWithContextPool<CrmDbSession>(p => p.GetRequiredService<ICrmConfig>().ConnectionString);
    } else if (key.StartsWith("postgres")) {
        services.AddConfig<ICrmConfig, Crm.Models.Postgres.CrmConfig>();
        services.AddSingleton<IExecuteScriptFile, ExecuteScriptFile>(); // generic SQL executor
        services.AddScoped(p => new CrmPostgresMigration(p.GetRequiredService<ICrmConfig>().ConnectionString));
        services.AddPostgresWithContextPool<CrmDbSession>(p => p.GetRequiredService<ICrmConfig>().ConnectionString);
    }
    services.RegisterCommands();
}
```

Use `ExecuteSqlServerScriptFile` (handles `GO` batch separators) for SQL Server, and `ExecuteScriptFile` for PostgreSQL.

### Deployment scripts convention

`ExecuteDeploymentScripts<T>` looks for `.sql` files in a subdirectory named after the application's semantic version (pre-release and metadata stripped), inside the `--directory` path you pass. Files are executed in alphabetical order.

```
deploy/
  1.2.0/
    001_add_index.sql
    002_seed_data.sql
```

If `--pre` is passed, the scripts are only executed when there are pending EF model changes (i.e., new migrations). Use this for pre-migration data transformations.

## Quick Checklist

When adding a new entity/feature, follow this order:

1. **Entity class + EntityMap** — define properties, key, indexes, relationships
2. **Repository interface + class** — query methods with eager loads; throw `NotFoundException<T>` on key lookups
3. **Service interface + class** — business logic; returns entities; no SaveChanges calls
4. **DI registration** — add to the module extension method
5. **Application boundary** — controller/handler calls SaveChangesAsync and handles `SaveResults`
6. **Admin CLI** — add migration context classes and wire verbs if a new DB provider is introduced
